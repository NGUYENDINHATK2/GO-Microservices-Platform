Motivation

Why go-ms?

Most Go microservice examples online are either too simple (a single CRUD handler) or too abstract (theoretical architecture diagrams). There's a gap between "hello world" tutorials and production systems that handle real concerns like token rotation, RBAC, transactional consistency, and observability.

go-ms bridges that gap. It's a working, runnable system that implements DDD, CQRS, event-driven architecture, and cloud-native infrastructure — the same patterns used at companies processing millions of requests.

Philosophy

Design principles

DDD

Domain First

Business logic lives in the domain layer with zero external dependencies. The domain drives the architecture, not the framework.

CQ

CQRS Separation

Commands and queries are separated at the application layer. Write paths go through the bus, read paths are direct.

EV

Event-Driven

Domain events are persisted atomically via the outbox pattern and reliably published to Kafka. No dual-write problem.

SC

Security by Design

Token Family rotation, RBAC with scoped permissions, MFA support, device tracking, and audit logging built in from day one.

Journey

Development timeline

Jan 2026

Project inception

Initial monorepo setup with Go workspace, shared module, and basic Identity & Gateway services.

Jan 2026

Ticket Service & caching

Added Ticket domain with 3-layer cache (Ristretto → Redis SWR → PostgreSQL), singleflight, and distributed locking.

Feb 2026

Event-driven architecture

Implemented transactional outbox, Kafka integration, and domain event publishing across services.

Mar 2026

Identity refactor — 7 domains

Major refactor: 14 tables across Identity, Credential, MFA, Verification, Session, RBAC, and Audit domains. Token Family Pattern with reuse detection.

Mar 2026

Notification Service & public release

Added Notification service, documentation site, and prepared for open-source release.
